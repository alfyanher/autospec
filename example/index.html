Architecture

Overview

AutoSpec is a Node.js CLI tool that generates a living documentation suite for any codebase by scanning source files, filling markdown prompt templates with extracted context, and invoking claude -p to produce structured markdown output. It exists to eliminate the gap between code and documentation by automating generation at commit time, making docs self-maintaining rather than manually curated.

Data Flow

  1. Entry — bin/autospec.js calls cli.js:run() with raw process.argv arguments.
  2. Config — config.js:loadConfig() reads .autospec.yaml and deep-merges it with DEFAULT_CONFIG; initConfig() also auto-detects project name and type from package.json, Cargo.toml, or directory structure.
  3. Scan — scanner.js:scanProject() walks the file tree (respecting .gitignore and config ignore lists via the ignore package), reads up to 30 prioritized source files, extracts ES/CJS/Python import statements, identifies route files by pattern-matching content, and estimates token usage.
  4. Template fill — generator.js:generateDocs() iterates DOC_CONFIGS, reads each prompt template from prompts/, substitutes {{PLACEHOLDER}} variables with context data via fillTemplate(), and truncates at 100k characters if needed.
  5. Claude invocation — the filled prompt is written to a temp file in .autospec/ and passed to claude -p via stdin redirect (shell pipe on Windows, stdin redirect on Unix); output is captured with a 120-second timeout.
  6. Write — each document is written to .autospec/<name>.md; CLAUDE.md is written to the project root via the ../ escape in resolveOutputPath().
  7. Report — reporter.js:generateHTMLReport() reads the generated markdown files, converts them to HTML (including Mermaid diagram unwrapping and GitHub-style callout preprocessing), and writes .autospec/index.html.
  8. Hooks — on init, hooks.js:setupHooks() installs a post-commit shell script that re-runs autospec generate --changed --quiet in the background and stages updated docs.

Key Design Patterns

  • Command map — src/cli.js uses a plain COMMANDS object keyed by command name; adding a new command requires only a new entry and a function in the same file.
  • Template-based prompts — all prompt logic lives in prompts/*.md files; generator.js only substitutes placeholders. Changing tone, structure, or output format requires no JS changes.
  • Structured result objects — generateDocs() returns { generated, failures } so partial failures never abort the full run; callers decide how to surface errors.
  • Additive git hooks — hooks.js:writeWithBackup() appends to existing hooks rather than overwriting them, preserving any pre-existing hook scripts.
  • Path normalization — normalizePath() in scanner.js converts all backslashes to forward slashes before storing paths, preventing Windows path format from leaking into prompt templates.
  • Conditional doc generation — DOC_CONFIGS entries carry a conditional field (e.g., hasRoutes); filterDocs() skips docs whose condition the scanned context does not satisfy.
  • File prioritization — prioritizeFiles() in scanner.js sorts source files by a pattern list (entry points first, then routes, services, models) so the most architecturally significant files fill the context budget first.

Infrastructure & Deployment

  • Claude Code CLI (claude -p) — required runtime dependency; invoked as a subprocess per document; version-checked at startup via checkClaudeCLI().
  • GitHub Actions — .github/workflows/autospec.yml and .github/actions/autospec/action.yml provide a composite action for CI-driven generation; supports full and changed modes with optional auto-commit.
  • Git hooks — post-commit hook runs autospec generate --changed in the background after every commit on Unix systems; pre-push hook optionally warns when docs may be stale.
  • npm global install — distributed as autospec binary via bin.autospec in package.json; no build step required.

Extension Points

  • New generated document — add an entry to DOC_CONFIGS in src/generator.js (with id, output, prompt, and optionally conditional) and create the corresponding prompts/<name>.md template.
  • New CLI command — add a key/function pair to the COMMANDS map in src/cli.js; the function receives parsed flags and is responsible for its own spinner and output.
  • New scanned context — extend scanProject() in src/scanner.js to populate new fields on the context object, then add corresponding {{PLACEHOLDER}} entries in fillTemplate() in src/generator.js.
  • New config option — add a key with a sensible default to DEFAULT_CONFIG in src/config.js; deep merge ensures existing user configs remain valid.
  • New prompt placeholder — add the key/value pair to the replacements object in fillTemplate(); use it in any prompt template with {{KEY}} syntax.
  • Custom project type detection — extend the detectProjectType() heuristics in src/config.js by adding directory name checks to the entrySet conditionals.

Project Conventions

  • ES Modules only — every file uses import/export; no require(). Example: src/cli.js opens with import { scanProject } from './scanner.js'
  • async/await everywhere — no raw Promise chains or callbacks. Example: generator.js uses await for every claude -p subprocess call
  • camelCase for variables and functions, UPPER_SNAKE for module-level constants — Example: DOC_CONFIGS array in src/generator.js
  • No inline prompts — all Claude prompt templates live as .md files under prompts/, never embedded in JS
  • Path normalization — all file paths are run through normalizePath() before storing to keep Windows backslashes from breaking templates
  • No process.exit() inside src/ — only src/cli.js is allowed to exit the process; all other modules throw or return errors
  • Scanner and generator are decoupled — src/scanner.js returns a plain context object and knows nothing about prompts or output files

Common Pitfalls

  • claude CLI not on PATH — AutoSpec shells out to claude -p; if it isn't installed and authenticated the run fails immediately. Install with npm install -g @anthropic-ai/claude-code and run claude login.
  • Running generate before init — AutoSpec requires .autospec.yaml at the project root. Always run init first; generate alone will error with "No .autospec.yaml found".
  • Git hooks silently skipped on Windows — Hook installation uses chmod, which isn't available on Windows. Hooks will be skipped without error; use the GitHub Action (.github/workflows/autospec.yml) for automatic updates instead.
  • Large repos timing out — Very large codebases can exceed Claude's context window. Add noisy directories to the ignore list in .autospec.yaml to reduce token usage.
  • Editing prompt output by hand — The .autospec/*.md files are regenerated on every run and will overwrite manual edits. Customize output by editing the corresponding file in prompts/ instead.
  • Confusing the demo project with AutoSpec itself — scripts/demo-project/ is a standalone Express API used only for recording demos. It has its own package.json; running npm install there installs its dependencies, not AutoSpec's.

Architectural Decisions

All decisions are inferred from static analysis. Status is estimated, not authoritative.

ADR-001: Node.js as Runtime

Field Value
Status active
Decided unknown
Evidence package.json — "engines": { "node": ">=18.0.0" }, "type": "module"

Context: A CLI tool for generating documentation needed a runtime with strong file I/O, process spawning, and cross-platform npm distribution.

Decision: Node.js 18+ was chosen as the sole runtime, distributed via npm as a global CLI.

Alternatives not chosen: Deno, Bun, Python, Go

Trade-offs: npm ecosystem and broad developer familiarity at the cost of a heavier install footprint compared to a compiled binary; Node 18 enables native ES Modules and the built-in test runner without extra dependencies.

ADR-002: ES Modules Over CommonJS

Field Value
Status active
Decided db0b501 (initial implementation)
Evidence package.json — "type": "module", all source files use import/export

Context: Modern Node.js projects benefit from static analysis, top-level await, and alignment with browser module standards. The project had no legacy CJS constraints.

Decision: Pure ES Modules (import/export) with no CommonJS require() anywhere in source.

Alternatives not chosen: CommonJS, mixed CJS/ESM with dual packaging

Trade-offs: Static imports are used everywhere; a single dynamic import() remains in src/cli.js for the getChangedFiles helper. This keeps module resolution predictable and enables tree-shaking across the codebase.

ADR-006: claude CLI subprocess as Generation Engine

Field Value
Status active
Decided db0b501 (initial implementation)
Evidence src/generator.js — execSync(cmd, ...) calling claude -p, checkClaudeCLI(), CLAUDE_TIMEOUT_MS

Context: AutoSpec needs to call a large language model to generate documentation from source context. Building a direct API integration would require bundling an API client and managing authentication.

Decision: AutoSpec shells out to the claude CLI using execSync, passing a filled prompt template via stdin pipe.

Alternatives not chosen: Direct Anthropic API via SDK, OpenAI API, local LLM via Ollama

Trade-offs: Delegates authentication and model selection to the already-installed Claude Code CLI, but creates a hard runtime dependency on a separately-installed tool and limits parallelism to one doc at a time due to synchronous execSync calls.

ADR-007: Markdown Prompt Templates in prompts/ Directory

Field Value
Status active
Decided db0b501 (initial implementation)
Evidence prompts/ directory with architecture.md, decisions.md, onboarding.md, etc.; CLAUDE.md — "never inline prompts in JS code"

Context: Prompt content evolves separately from generation logic. Inlining prompts in JavaScript makes them hard to read, edit, and contribute to.

Decision: Each generated document type has a corresponding .md file in prompts/ loaded at runtime by generator.js.

Alternatives not chosen: Prompts embedded in JS strings, a separate prompt registry JSON file, a database of prompts

Trade-offs: Prompts are editable without touching JS code and can be contributed to independently, but require a file read per document generation and a defined {{PLACEHOLDER}} contract between templates and fillTemplate() in generator.js.

ADR-008: YAML for Configuration

Field Value
Status active
Decided db0b501 (initial implementation)
Evidence src/config.js — import YAML from 'yaml', .autospec.yaml config file, package.json dependency "yaml": "^2.3.0"

Context: AutoSpec needs a user-editable configuration file. JSON lacks comments; TOML has weaker Node.js ecosystem support. YAML is the dominant format for developer tooling config (GitHub Actions, Docker Compose).

Decision: .autospec.yaml is the project configuration file, parsed with the yaml npm package.

Alternatives not chosen: JSON (.autospec.json), TOML, JavaScript config file (.autospec.js)

Trade-offs: YAML supports comments and is familiar to developers, but has edge-case parsing pitfalls (e.g., ambiguous booleans, indentation sensitivity). The yaml package adds a small dependency but is well-maintained.

ADR-009: Post-Commit Git Hook for Automatic Updates

Field Value
Status active
Decided db0b501 (initial implementation)
Evidence src/hooks.js — HOOK_CONTENT post-commit script, setupHooks(), autospec generate --changed --quiet &

Context: Documentation becomes stale if it requires manual regeneration after every change. Automating regeneration at commit time ensures docs stay current without developer effort.

Decision: A post-commit git hook is installed that runs autospec generate --changed --quiet in the background with a 30-second timeout.

Alternatives not chosen: pre-commit hook, CI-only updates (GitHub Action), file watcher (chokidar), pre-push hook only

Trade-offs: Post-commit runs non-blocking in the background and stages updated docs automatically, but is limited to Unix (hooks require chmod). Windows users must use the GitHub Action. The hook appends rather than overwrites to preserve any existing hook scripts.

ADR-010: Error Middleware for Centralized HTTP Error Handling (Demo Project)

Field Value
Status active
Decided unknown
Evidence scripts/demo-project/src/middleware/errors.js — errorHandler function, scripts/demo-project/src/index.js — app.use(errorHandler) registered last

Context: Validation errors from Zod and unexpected server errors needed consistent JSON response formatting without duplicating error logic in every route handler.

Decision: A single Express error middleware (errorHandler) is registered last in the middleware stack to intercept ZodError and generic errors and format them as JSON responses.

Alternatives not chosen: Per-route try/catch with inline error responses, a dedicated error class hierarchy, express-async-errors patch

Trade-offs: Centralizes error formatting in one place and makes route handlers cleaner, but requires that all async route errors propagate via next(err) or thrown exceptions rather than being silently swallowed.

Boundary Concerns

  • Generator → claude subprocess: Generator is tightly coupled to the claude -p CLI contract (flags, exit codes, stdout format). Any change to the Claude CLI interface breaks generation silently unless the subprocess call is wrapped with explicit error checking.
  • Generator → Reporter: Reporter is not independently reachable — it can only be called through Generator. If report writing ever needs to be triggered from CLI Core directly (e.g., for a --report-only flag), this boundary will need renegotiation.
  • Demo Project isolation: The demo project has its own package.json and dependency tree but shares the repo root .gitignore. It is effectively an embedded sub-project with no formal workspace configuration, which could cause confusion if npm install is run from the root.
  • src/scanner.js import map oddity: The import map shows y, y, y, y as imports alongside ignore — these appear to be parsing artefacts rather than real dependencies. No y package exists in package.json; the actual named imports from yaml or internal helpers may have been collapsed during context extraction.
